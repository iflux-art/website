# 贡献指南

感谢您对斐流艺创项目的关注！我们欢迎各种形式的贡献，包括代码、文档、设计和测试等。本文档将指导您如何为项目做出贡献。

## 贡献方式

### 1. 报告问题

如果您发现了 bug 或有功能建议，请通过以下方式提交：

1. **检查是否已存在**：在提交新问题前，请先搜索 [GitHub Issues](https://github.com/iflux-art/official/issues) 确认问题未被报告
2. **创建 Issue**：如果问题不存在，请创建一个新的 Issue
3. **提供详细信息**：包括问题描述、复现步骤、环境信息等

### 2. 提交代码

#### 开发流程

1. **Fork 项目**：
   - 访问 [项目主页](https://github.com/iflux-art/official)
   - 点击右上角的 "Fork" 按钮

2. **克隆仓库**：
   ```bash
   git clone https://github.com/your-username/official.git
   cd official
   ```

3. **创建功能分支**：
   ```bash
   git checkout -b feature/your-feature-name
   ```

4. **安装依赖**：
   ```bash
   pnpm install
   ```

5. **开发功能**：
   - 遵循[开发规范](/docs/development/development-standards)
   - 编写必要的测试
   - 更新相关文档

6. **运行检查**：
   ```bash
   # 运行代码检查
   pnpm check
   
   # 运行测试
   pnpm test
   
   # 检查类型
   pnpm type-check
   ```

7. **提交代码**：
   ```bash
   git add .
   git commit -m "feat: add your feature description"
   git push origin feature/your-feature-name
   ```

8. **创建 Pull Request**：
   - 访问您的 Fork 仓库
   - 点击 "Compare & pull request" 按钮
   - 填写 PR 描述并提交

### 3. 改进文档

文档是项目的重要组成部分，您可以通过以下方式改进文档：

1. **修正错误**：修正文档中的拼写错误、语法错误或技术错误
2. **补充内容**：添加缺失的说明或示例
3. **翻译文档**：将文档翻译成其他语言
4. **优化结构**：改进文档的组织结构和导航

### 4. 设计贡献

如果您擅长 UI/UX 设计，可以通过以下方式贡献：

1. **界面优化**：提出界面改进建议
2. **设计规范**：完善设计系统和组件规范
3. **图标设计**：设计新的图标或优化现有图标
4. **用户体验**：改善用户交互流程

## 代码贡献规范

### 1. 开发环境设置

请确保您的开发环境满足以下要求：

- Node.js >= 22.x
- pnpm >= 9.11.0
- Git

安装依赖：

```bash
pnpm install
```

### 2. 代码质量要求

在提交代码前，请确保：

1. **通过代码检查**：
   ```bash
   pnpm check
   ```

2. **通过测试**：
   ```bash
   pnpm test
   ```

3. **类型检查通过**：
   ```bash
   pnpm type-check
   ```

4. **代码格式化**：
   ```bash
   pnpm format
   ```

### 3. 提交信息规范

遵循 [Conventional Commits](https://www.conventionalcommits.org/) 规范：

```bash
# 功能新增
git commit -m "feat: add user authentication system"

# 问题修复
git commit -m "fix: resolve login timeout issue"

# 文档更新
git commit -m "docs: update API documentation"

# 代码重构
git commit -m "refactor: optimize component structure"

# 性能优化
git commit -m "perf: improve loading speed"

# 测试相关
git commit -m "test: add unit tests for search functionality"

# 构建相关
git commit -m "build: update dependencies"
```

### 4. Pull Request 要求

#### PR 标题

使用与提交信息相同的格式：

```
feat: add user authentication system
fix: resolve login timeout issue
```

#### PR 描述

PR 描述应包含以下内容：

1. **变更描述**：简要说明本次变更的内容
2. **解决的问题**：关联的 Issue 或问题描述
3. **测试方法**：如何测试这些变更
4. **截图**（如果适用）：UI 变更的截图

#### 代码审查

PR 需要至少一名维护者审查通过后才能合并。

## 文档贡献规范

### 1. 文档结构

文档采用 [Diátaxis](https://diataxis.fr/) 框架组织：

1. **教程 (Tutorials)**：学习导向的动手指南
2. **操作指南 (How-to Guides)**：问题导向的实用指南
3. **技术参考 (Reference)**：信息导向的技术文档
4. **解释说明 (Explanation)**：理解导向的概念说明

### 2. 文档格式

使用 MDX 格式编写文档，遵循以下规范：

1. **Frontmatter**：
   ```mdx
   ---
   title: 文档标题
   description: 文档描述
   date: 2025-08-29
   category: 文档分类
   tags:
     - 标签1
     - 标签2
   ---
   ```

2. **标题层级**：
   - 使用 `#` 作为主标题
   - 使用 `##` 作为章节标题
   - 使用 `###` 作为子章节标题

3. **代码示例**：
   ```mdx
   ```bash
   # 命令示例
   pnpm dev
   ```
   
   ```typescript
   // TypeScript 代码示例
   const greeting = (name: string): string => {
     return `Hello, ${name}!`;
   };
   ```
   ```

### 3. 文档更新

当代码变更影响到文档时，请同步更新相关文档：

1. **API 变更**：更新 API 文档
2. **功能新增**：添加新的使用指南
3. **配置变更**：更新配置说明

## 测试贡献规范

### 1. 测试类型

项目包含以下测试类型：

1. **单元测试**：测试单个函数或组件
2. **集成测试**：测试多个组件的交互
3. **端到端测试**：测试完整的用户流程

### 2. 测试工具

- **Vitest**：单元测试框架
- **React Testing Library**：React 组件测试工具
- **Playwright**：端到端测试工具（如需要）

### 3. 测试覆盖率

目标测试覆盖率达到 80% 以上：

```bash
# 运行测试并生成覆盖率报告
pnpm test:coverage
```

### 4. 测试编写规范

1. **描述性测试名称**：
   ```typescript
   it('should format date correctly', () => {
     // 测试实现
   });
   ```

2. **独立测试**：
   - 每个测试应独立运行
   - 避免测试间的依赖关系

3. **边界条件测试**：
   - 测试正常情况
   - 测试异常情况
   - 测试边界条件

## 社区行为规范

### 1. 尊重他人

- 使用包容性语言
- 尊重不同观点和经验
- 专注于技术讨论而非个人攻击

### 2. 建设性反馈

- 提供具体、可操作的建议
- 解释为什么需要修改
- 承认贡献者的努力

### 3. 透明沟通

- 清晰表达自己的观点
- 及时回应他人的问题
- 承认自己的错误
